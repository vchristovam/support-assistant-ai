1.  Project Manifesto
    Project Name: Enterprise Support Autopilot Goal: Automate Tier-2 technical support tasks (data queries, health checks, documentation retrieval) using a modular, secure, multi-agent AI system. Core Philosophy: "Specialists over Generalists." The system uses a Supervisor to route tasks to isolated sub-agents. Features must be "plug-and-play"—removing a feature (e.g., Azure RAG) should only require deleting its folder and one line of registration code.
2.  Technical Stack
    • Runtime: Node.js (LTS v20+)
    • Orchestration: LangChain.js & LangGraph.js
    • Server: Fastify or Express (for SSE streaming)
    • Persistence: Redis (@langchain/langgraph-checkpoint-redis) or Postgres for production state.
    • Integrations:
    ◦ Databricks: @databricks/sql (SQL execution) & Genie Conversation API (Natural Language).
    ◦ Dynatrace: Dynatrace Environment API v2 (DQL, Problems).
    ◦ Azure: @azure/search-documents & @langchain/azure-openai.
3.  High-Level Design (Supervisor Pattern)

stream.ts # Server-Sent Events (SSE) logic 5. Component Specifications
A. The Supervisor (Router)
• Logic: An LLM node that assesses conversation history and outputs a structured tool call to transfer control to a worker.
• State: Uses MessagesAnnotation to preserve the full chat history.
• Handoff: Uses create_handoff_tool pattern.
◦ Prompt: "You are a support triage lead. Analyze the request. If it concerns database data, call transfer_to_databricks. If it concerns system errors/alerts, call transfer_to_dynatrace..."
B. Databricks Worker (Data Context)
This agent handles two distinct interaction modes:

1. Structured SQL (Deterministic): Uses @databricks/sql driver to execute read-only queries (SELECT) for precise data retrieval.
2. Genie (Exploratory): Uses the Databricks Genie API.
   ◦ Flow: The agent calls POST /api/2.0/genie/spaces/{space_id}/start-conversation to ask questions like "Why did orders drop yesterday?" and returns the text/SQL generated by Genie.
   C. Dynatrace Worker (Observability)
   • Tools:
   ◦ get_problems: Fetches active alerts from /api/v2/problems.
   ◦ execute_dql: Runs Dynatrace Query Language (DQL) queries to search logs/metrics.
   • Safety: Read-only API tokens.
   D. Operations Worker (The HITL Layer)
   This is the only agent allowed to change state (e.g., cancel order).
   • Mechanism: 1. Agent calls tool request_order_cancellation. 2. Graph executes interrupt() with payload { type: "approval_required", action: "cancel_order", data: ... }. 3. Execution halts. State is saved to Redis. 4. API Response: Sends a special event to frontend: "Waiting for approval". 5. Resume: Admin clicks "Approve" -> API calls graph.invoke(new Command({ resume: "approved" })).
   E. Human Interface Worker (User Clarification)
   • Purpose: Ask clarifying questions when information is missing
   • Mechanism: 1. Supervisor hands off to human_interface_agent 2. Agent calls ask_human tool which triggers interrupt() 3. Graph pauses, streams question to frontend 4. User submits answer via POST /chat/answer 5. Graph resumes with Command({ resume: answer }) 6. Answer returned to Supervisor
   F. Health Check Worker (System Monitoring)
   • Purpose: Monitor system health, diagnose issues, check service status
   • Tools:
   ◦ check_integration_health: Check Databricks, Dynatrace, Azure, Redis status
   ◦ check_internal_service: Check B3 calculator, order processing, etc.
   ◦ get_system_metrics: Get performance, errors, resource metrics
   ◦ check_endpoint: Check specific HTTP endpoints
   • Use Cases:
   - "Calculator slow" → check_internal_service("B3 calculator")
   - "System status" → get_system_metrics("all")
   - "Is Databricks down?" → check_integration_health("databricks")
     G. Persistence Layer
     • Technology: RedisSaver (from @langchain/langgraph-checkpoint-redis).
     • Why: Low latency (<1ms) is critical for chat interfaces.
     • Configuration: Enable pickle_fallback if storing complex objects, though JSON-serializable state is preferred.

   Copy/paste these specific instructions to your coding assistant to build the core:

   Step 2: Build the Databricks Tooling

   Step 3: Implement HITL Logic
   "In the Operations Agent, wrap the modify_data tool execution with an interrupt. If the tool is called, pause execution and return the tool call details to the client. Allow the graph to be resumed with a string 'APPROVED' or 'REJECTED'."
   Step 4: The Streaming Server
   "Set up an Express route /chat that initializes the graph with a RedisSaver. Use graph.streamEvents to stream output. Format the stream as Server-Sent Events (SSE) so the frontend can display tokens in real-time."
4. How to Remove a Feature (e.g., Removing Azure RAG)
   To ensure the "MVP" requirement of modularity:
5. Delete /src/agents/knowledge.
6. Update /src/graph/index.ts: Remove the import of the knowledge agent and remove the .addNode("knowledge_agent", ...) line.
7. Update /src/agents/supervisor/prompts.ts: Remove the instruction "For documentation, use the knowledge agent."
8. Result: The graph recompiles, the node disappears, and the Supervisor stops routing to it. No other code breaks.

   To remove the Human Interface feature:
   1. Delete /src/agents/human_interface
   2. Update /src/agents/supervisor/index.ts: Remove import and transferToHumanInterface tool
   3. Update /src/agents/supervisor/prompts.ts: Remove human_interface_agent mention
   4. Remove /chat/answer endpoint from /src/server/app.ts

   To remove the Health Check feature:
   1. Delete /src/agents/health_check
   2. Update /src/agents/supervisor/index.ts: Remove import and transferToHealthCheck tool
   3. Update /src/agents/supervisor/prompts.ts: Remove health_check_agent mention

## 5. Agent Chat UI Integration

The Enterprise Support Autopilot implements the LangGraph Platform API specification for compatibility with the Agent Chat UI - a modern, responsive chat interface built by LangChain.

### 5.1 Overview

The integration provides:

- **Real-time streaming** via Server-Sent Events (SSE)
- **Thread management** with persistent conversation history
- **Human-in-the-Loop** approval workflows
- **Multi-agent routing** through a visual interface

### 5.2 API Compatibility Layer

The Support Server (`/src/server/app.ts`) exposes LangGraph Platform-compatible endpoints:

```typescript
// Thread Management
POST / api / threads; // Create thread
GET / api / threads / { id }; // Get thread
GET / api / threads / { id } / history; // Get messages

// Run Execution
POST / api / threads / { id } / runs; // Create run
GET / api / threads / { id } / runs / { run_id }; // Get run status
POST / api / threads / { id } / runs / { run_id } / cancel; // Cancel run

// SSE Streaming
GET / api / threads / { id } / runs / { id } / stream; // Stream events

// HITL Interrupts
POST / api / threads / { id } / runs / { run_id } / interrupt; // Resume from interrupt
```

These endpoints follow the LangGraph Platform specification exactly, enabling seamless integration with Agent Chat UI without modification.

### 5.3 Thread and Run Management

**Thread Lifecycle:**

1. Client creates thread via `POST /api/threads`
2. Thread state persisted to Redis (or MemorySaver in dev)
3. Client creates run via `POST /api/threads/{id}/runs`
4. Run executes graph with provided input
5. State checkpointed after each node

**State Persistence:**

```typescript
// Using Redis for production
const checkpointer = await createRedisCheckpointer(config.redis.url);

// Using MemorySaver for development
const checkpointer = new MemorySaver();
```

### 5.4 SSE Streaming Format

The server streams execution events in standard SSE format:

```
event: metadata
data: {"run_id":"run-...","thread_id":"thread-..."}

event: values
data: {"messages":[...],"next_node":"..."}

event: messages
data: [{"type":"ai","content":"partial"}]

event: events
data: {"event":"on_chat_model_stream",...}

event: end
data: {}
```

Stream modes supported:

- `values` - Complete state after each step
- `messages` - Streaming message chunks
- `events` - Raw LangChain events
- `updates` - State deltas

### 5.5 Interrupt Schema

When a worker requires human input, the graph interrupts with a standardized schema:

```typescript
interface HITLInterruptValue {
  type: "approval" | "input" | "selection";
  description: string;
  value?: unknown;
  actions?: Array<{
    id: string;
    label: string;
    type: "approve" | "reject" | "edit";
  }>;
  schema?: object; // JSON Schema for validation
}
```

**Example - Approval Request:**

```json
{
  "type": "approval",
  "description": "Approve database schema modification",
  "value": {
    "action": "modify_table",
    "table": "users",
    "changes": ["add column: email_verified"]
  },
  "actions": [
    { "id": "accept", "label": "Approve", "type": "approve" },
    { "id": "reject", "label": "Reject", "type": "reject" },
    { "id": "edit", "label": "Edit", "type": "edit" }
  ]
}
```

**Resuming from Interrupt:**

```bash
POST /api/threads/{thread_id}/runs/{run_id}/interrupt
{
  "action": "accept",
  "value": null
}
```

### 5.8 Deployment Considerations

**Production Requirements:**

- Redis for state persistence (required)
- SSL/TLS termination
- Load balancer with SSE support (disable buffering)
- Horizontal scaling supported via Redis

**Load Balancer Configuration:**

```nginx
location / {
    proxy_pass http://backend;
    proxy_buffering off;        # Critical for SSE
    proxy_read_timeout 86400;   # Long-lived connections
}
```
